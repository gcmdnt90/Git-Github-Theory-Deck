Same git. Two ways to drive it. You should understand both, because you stay responsible either way.

Module 1

Why Git is worth it

Before the commands: what problem does it actually solve, and why do people who use it never go back?

Motivation first. If people feel the pain git removes, the commands later feel worth learning.

The problem without Git

You know this file folder:

report_final.docx
report_final_v2.docx
report_final_REAL.docx
report_final_use-this-one.docx

It breaks down the moment you need to answer: what changed, when, and can I get the old version back?

And with two people editing, it gets worse: whose copy is right? Which changes got lost when you emailed versions back and forth?

Manual versioning fails silently. You only notice when something is already lost.
Everyone recognises the "_final_final" pattern. This is the emotional hook.

What Git gives you instead

A time machine

Every saved snapshot is recoverable. Broke something an hour ago? Go back to exactly how it was.

A safety net

Experiment freely. If it goes wrong, you throw the change away — the last good version is untouched.

A team workflow

Many people change the same project at once, and Git merges the work and flags real conflicts.

Git turns “don’t touch it, it works” into “try anything, we can always go back.”
Three benefits: recover, experiment safely, collaborate. Confidence to change things is the real payoff.

Why it matters more in the age of agents

An AI agent can change dozens of files in seconds. That is powerful — and risky if you can’t see or undo what it did.

Git is what makes agent help safe to accept: every change is a reviewable snapshot you can inspect, keep, or throw away.

With Git: “Show me exactly what you changed” and “undo that” are one command away.
Without Git: an agent’s edits overwrite your files with no record and no undo.
Git is the seatbelt for working with agents. It doesn’t slow you down — it lets you go faster safely.
Tie motivation to the deck's theme: agents make version control non-optional, because they change a lot fast.

Module 2

The mental model

Six words unlock almost everything in Git. Learn what they mean and the commands stop feeling random.

The single biggest reason git feels confusing is people learn commands before the model. Fix that here.

The six words

Repository (repo)

Your project folder, plus its entire history. Created once with git init or by cloning.

Working directory

The actual files as they are right now on your computer, where you edit.

Staging area

A holding tray. You pick which changes go into the next snapshot before you commit.

Commit

A saved snapshot with a message. The permanent, recoverable unit of history.

Branch

A movable line of commits. Lets you work on something without touching the main version.

Remote

A copy of the repo somewhere else — usually GitHub — that you push to and pull from.

Read them once slowly. Staging is the one newcomers miss; it's why commit is two steps (add, then commit).

How a change travels

A change moves through four places. Each git command just moves it one step along:

Working dir you edit a file
Staging git add
Local repo git commit
Remote (GitHub) git push

And it comes back the other way with git pull when someone else pushed changes.

Almost every command you’ll learn is just “move this change one box to the right (or left).”
This diagram is the spine of the whole deck. Point at it again in module 3. add = stage, commit = save locally, push = send to GitHub, pull = receive from GitHub.

Does the agent need a different model? No.

When you ask an agent to “save this and push it,” it walks the exact same four boxes: it stages, commits, and pushes.

So this one diagram lets you check the agent’s work. If it says “committed and pushed,” you know precisely what that means and where your change now lives.

Understanding the model is what turns you from “hoping it worked” into “knowing it worked.”
Reinforce: the mental model is the shared language between you and the agent. It's not extra work, it's leverage.

Module 3

The everyday loop

Five commands cover 90% of daily Git. Learn these and you can work for real.

Be honest: this is the module people actually need. Everything else is occasional. Drill the loop.

The loop, once

You repeat this cycle all day: see what changed, stage it, save it, share it.

git statusWhat changed and what’s staged. Run it constantly — it’s free.
git add <file>Put a change in the staging tray (git add . for everything).
git commit -m "message"Save a snapshot of the staged changes, with a note.
git pushSend your commits up to GitHub.
git pullBring down commits others pushed. Do this before you start.
status → add → commit → push. Then pull to stay in sync. That’s the job.
Two habits to preach: run status often, and pull before you begin a work session.

Saving your work — both ways

You (human) 
git add .
git commit -m "Add intro section"
git push

Three commands: stage everything, snapshot it with a message, send it to GitHub.

Ask the agent

“Commit the changes with a sensible message and push.”

The agent runs the same three commands. It usually writes the commit message for you — read it before approving.

GUI equivalent: VS Code’s Source Control panel and GitHub Desktop do stage / commit / push with buttons — same three steps.

Point out the message is the one thing worth reviewing on the agent side. Everything else is mechanical.

Seeing what’s going on — both ways

You (human) 
git status      # what changed
git diff        # the exact line edits
git log --oneline   # history so far
Ask the agent

“What’s changed since my last commit?”

“Summarise the last five commits.”

The agent reads the same output and explains it in plain language.

status = what, diff = which lines, log = history. These three answer “what’s happening?”
diff is the reviewing tool. It's how you check both your own and the agent's changes before committing.

Starting a repo (you do this once)

New project on your machine:

git init            # make this folder a repo
git add .
git commit -m "First commit"

Existing project from GitHub:

git clone https://github.com/user/project.git

Clone downloads the repo and its full history, ready to work.

Or just tell the agent: “Initialise a git repo here” / “Clone this GitHub repo for me.”

init vs clone is the one-time setup. After that it's the daily loop forever.

Module 4

Branching & merging

How to try something new without risking the version that already works.

Branching is what makes git feel safe for teams and for experiments. Keep it concrete.

What a branch is for

Your working version lives on a branch, usually called main. You want main to always work.

To try a change, you branch off: make a parallel copy, experiment there, and only bring it back to main when it’s good.

If the experiment fails, you just delete the branch. main never knew it existed.

This is why teams can work in parallel: everyone on their own branch, merging in when ready.

The mental image: main is the trunk, branches are side experiments that either merge back or get thrown away.

Branching — both ways

You (human) 
git switch -c new-feature   # create + move to it
# ...edit, add, commit as usual...
git switch main             # go back
git merge new-feature       # bring the work in
Ask the agent

“Make a branch called new-feature and work there.”

“Merge new-feature back into main.”

The agent runs the same switch / merge commands.

git switch is the modern command; you may also see git checkout do the same job.

Create-and-switch is one flag: -c. Merge happens from the branch you want to receive the changes (main).

Merge conflicts — not scary

A conflict happens when two branches changed the same lines. Git can’t guess which to keep, so it asks you.

It marks the spot in the file:

<<<<<<< your version
hello world
=======
hello there
>>>>>>> their version

You edit the file to the version you want, delete the marker lines, then git add and git commit.

Ask the agent

“Resolve this merge conflict, keeping both changes where sensible, and explain what you did.”

A conflict is just Git asking a question. Agents are good at proposing a resolution — you confirm it’s right.
Demystify conflicts: they're normal, they're local, nothing is lost. This is a great task to delegate then review.

Module 5

GitHub

Git lives on your computer. GitHub is the shared home online where teams meet.

Draw the line clearly: Git = the tool, GitHub = a website that hosts Git repos and adds collaboration features.

Git vs GitHub — the one distinction people mix up

Git

The version-control tool on your machine. Works fully offline. Made by no single company.

GitHub

A website that hosts Git repos in the cloud and adds sharing, review, and issue tracking. (GitLab and Bitbucket are alternatives.)

You can use Git with no GitHub. GitHub is where Git repos go to be shared and collaborated on.
Analogy: Git is like a camera (takes snapshots); GitHub is like a shared photo album online. Optional but very common.

The GitHub connection

Your local repo talks to GitHub through two commands you already met:

git clone <url>Copy a GitHub repo (with history) down to your machine.
git pushSend your local commits up to GitHub for others to see.
git pullBring down what others pushed. Do this before starting work.
Golden habit: pull before you start, push when you pause. It keeps you and your team in sync and avoids most conflicts.
clone once, then push/pull forever. The pull-first habit prevents the majority of merge headaches.

Pull requests: how changes get reviewed

A pull request (PR) is how you propose merging your branch into main on GitHub. It says: “here’s my work — please review before it goes in.”

Teammates comment, request changes, and approve. Then the branch is merged through the GitHub button.

Also on GitHub:

Issues a to-do / bug list for the project.
Fork your own copy of someone else’s repo to build on.
README the front-page description of the repo.

A pull request is a review checkpoint. It’s the same “human approves before it counts” idea, for teams.
PR = propose + review + merge. Connect it to the deck's theme: review before it counts, whether the author was a human or an agent.

Working with GitHub — both ways

You (human) 
git clone https://github.com/team/app.git
git switch -c fix-typo
# edit, commit
git push -u origin fix-typo

Then open a pull request on the GitHub website.

Ask the agent

“Clone team/app, make a branch, fix the typo, push it, and open a pull request.”

Agents with GitHub access can do the whole flow — you review the PR before merging.

Pushing usually requires a one-time login (a token or SSH key). An agent may ask you to set this up once.

Note the auth reality: first push needs credentials. This is the one setup snag beginners hit.

Module 6

Working with agents

An agent can run every Git command for you. The skill that stays yours: deciding what to accept.

This is the module that makes this deck different from a normal git tutorial. Emphasise judgement, not commands.

What’s worth delegating

Great to hand off low risk

Writing commit messages · staging and committing · summarising history · drafting a .gitignore · explaining a diff · proposing a conflict resolution.

Keep your hands on high risk

Force-pushing · deleting branches · rewriting history · anything touching shared main · anything involving credentials or secrets.

Delegate the typing. Own the decisions that are hard to undo.
The split is by reversibility. Reversible + tedious = delegate. Irreversible or shared = you decide.

The review habit

Before you accept an agent’s Git work, do the same check you’d do for a colleague’s:

1. Read the diff

“Show me exactly what changed.” Skim the actual lines, not just the summary.

2. Check the message

Does the commit message honestly describe the change? Rewrite it if not.

3. Confirm the target

Which branch? Pushed where? Never let a surprise land on shared main.

Git makes this safe: because every agent action is a commit, you can always inspect it and undo it.
Three-step review. Reading the diff is non-negotiable. This is where the git mental model pays off directly.

Good prompts for Git tasks

Clear

“Stage only the changes in report.md, commit with message ‘Add methods section’, and show me the diff before pushing.”

Too vague

“Fix the git stuff and sort it out.”

Vague prompts invite the agent to guess — and Git guesses can be hard to reverse.

Name the files, name the message, and ask to see the diff before anything is pushed.
Precision plus "show me before you push" is the safest delegation pattern. Model it live if presenting.

You stay responsible

Your name goes on the commit. Your team relies on what lands in main. The agent is a fast pair of hands — the accountability is still yours.

This is why the earlier modules matter: you can only review what you understand. The mental model is what makes delegation safe.
Close the module by tying back to the thesis slide: agent does the work, human reviews before it counts.

Module 7

Rules to never forget

The handful of habits and warnings that separate confident Git users from stressed ones.

This is the "if you remember nothing else" module. Practical safety, not theory.

Good habits

Commit small & often

One logical change per commit. Small commits are easy to understand and easy to undo.

Write real messages

“Fix login crash on empty email” beats “stuff”. Future-you reads these.

Pull before you start

Sync with the team first. It prevents most merge conflicts.

Small commits + honest messages + pull first = 90% of a smooth Git life.
These three habits prevent most pain. Commit granularity is the one people underdo.

Two things that bite

Never commit secrets. Passwords, API keys, tokens — once pushed, treat them as leaked even if you delete them later, because they stay in history.

Use a .gitignore file to tell Git which files to never track (keys, .env, large data, build junk).

Don’t force-push shared branches. git push --force can erase teammates’ work on main. If you ever need it, ask first and use it only on your own branch.
Secrets-in-history is the classic disaster. Force-push on shared main is the classic team disaster. Both are avoidable.

How to undo (the calm menu)

Almost nothing in Git is truly lost. Match the situation to the fix:

git restore <file>Throw away un-staged edits to a file — back to last commit.
git restore --staged <file>Un-stage something you added by mistake.
git commit --amendFix the last commit’s message or contents (before pushing).
git revert <commit>Safely undo an already-pushed commit by adding an opposite one.
Rule of thumb: revert for anything already shared (it’s safe); restore/amend for local, not-yet-pushed work.
revert vs reset: teach revert (safe, adds a commit) for shared history; avoid reset --hard for beginners.

Undoing — both ways

You (human) 
git log --oneline      # find the bad commit
git revert a1b2c3d      # undo it safely
Ask the agent

“My last commit broke things — revert it and explain what you did.”

The agent finds the commit and reverts it. You confirm the right one was undone.

When something breaks: don’t panic, don’t delete files. Ask “what changed?”, then revert.
The emotional message: git means mistakes are recoverable. Reverting is a normal daily action, not a failure.

The cheat sheet

One page to keep. If you internalise this slide, you can use Git for real work today.

This is the takeaway artefact. Tell people to screenshot it. Everything here appeared earlier in context.

The commands you actually use

Daily loop

git statuswhat changed
git add .stage everything
git commit -m "..."save a snapshot
git pushsend to GitHub
git pullget others’ work

Look & branch & undo

git diffwhich lines changed
git log --onelinehistory
git switch -c namenew branch
git merge namebring branch in
git revert <id>undo a pushed commit
Ten commands. This really is enough for the vast majority of daily work.

The rules, in one breath

Do: commit small & often · write honest messages · pull before you start · read the diff before you accept · use .gitignore.

Don’t: commit secrets · force-push shared branches · let an agent’s change land on main unreviewed · panic — almost nothing is truly lost.

Snapshots you can trust, branches to experiment, a remote to share — and a human who reviews before it counts.
The do/don't pairing is the safety summary. Close on the thesis sentence one more time.

Working with your agent, summed up

Your job

Understand the model. Decide what to change. Review the diff. Own what lands.

The agent’s job

Run the commands. Draft messages. Explain history. Propose fixes and conflict resolutions.

You bring judgement; the agent brings speed. Git is the shared, reviewable record between you.
Final framing. This division of labour is the whole point of the module.
