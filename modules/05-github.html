GitHub

Git lives on your computer. GitHub is the shared home online where teams meet.

Draw the line clearly: Git = the tool, GitHub = a website that hosts Git repos and adds collaboration features.

Git is the tool; GitHub is the shared place

Git

The version-control tool on your machine. Works fully offline and is not owned by GitHub.

GitHub

A website that hosts Git repos in the cloud and adds sharing, review, and issue tracking. (GitLab and Bitbucket are alternatives.)

You can use Git with no GitHub. GitHub is where Git repos go to be shared and collaborated on.
Analogy: Git is like a camera (takes snapshots); GitHub is like a shared photo album online. Optional but very common.

Your local repo and GitHub stay in sync

Your local repo talks to GitHub through three commands you already met:

git clone <url>Copy a GitHub repo (with history) down to your machine.
git pushSend your local commits up to GitHub for others to see.
git pullBring down what others pushed. Do this before starting work.
Local repoyour machine
GitHubshared remote
Golden habit: pull before you start, push when you pause. It keeps you and your team in sync and avoids most conflicts.
clone once, then push/pull forever. The pull-first habit prevents the majority of merge headaches.

Pull requests are review checkpoints

A pull request (PR) is how you propose merging your branch into main on GitHub. It says: “here’s my work — please review before it goes in.”

Teammates comment, request changes, and approve. Then the branch is merged through the GitHub button.

Also on GitHub:

Issues a to-do / bug list for the project.
Fork your own copy of someone else’s repo to build on.
README the front-page description of the repo.

A pull request is a review checkpoint. It’s the same “human approves before it counts” idea, for teams.
PR = propose + review + merge. Connect it to the deck's theme: review before it counts, whether the author was a human or an agent.

Ask the agent

“Clone team/app, make a branch, fix the typo, push it, and open a pull request.”

Agents with configured GitHub access can do the whole flow — you review the PR before merging.

Pushing usually requires a one-time login (a token or SSH key). An agent may ask you to set this up once.

Note the auth reality: first push needs credentials. This is the one setup snag beginners hit.
