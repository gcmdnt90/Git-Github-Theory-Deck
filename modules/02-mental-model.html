Module 2

The mental model

Six words unlock almost everything in Git. Learn what they mean and the commands stop feeling random.

The single biggest reason git feels confusing is people learn commands before the model. Fix that here.

Six Git words unlock the workflow

Repository (repo)

Your project folder, plus its entire history. Created once with git init or by cloning.

Working directory

The actual files as they are right now on your computer, where you edit.

Staging area

A holding tray. You pick which changes go into the next snapshot before you commit.

Commit

A saved snapshot with a message. The permanent, recoverable unit of history.

Branch

A movable line of commits. Lets you work on something without touching the main version.

Remote

A copy of the repo somewhere else — usually GitHub — that you push to and pull from.

Read them once slowly. Staging is the one newcomers miss; it's why commit is two steps (add, then commit).

Changes move through four places

A change moves through four places. Each git command just moves it one step along:

Working dir you edit a file
Staging git add
Local repo git commit
Remote (GitHub) git push

And it comes back the other way with git pull when someone else pushed changes.

Almost every command you’ll learn is just “move this change one box to the right (or left).”
This diagram is the spine of the whole deck. Point at it again in module 3. add = stage, commit = save locally, push = send to GitHub, pull = receive from GitHub.

Agents use the same four boxes

When you ask an agent to “save this and push it,” it walks the exact same four boxes: it stages, commits, and pushes.

So this one diagram lets you check the agent’s work. If it says “committed and pushed,” you know precisely what that means and where your change now lives.

Understanding the model is what turns you from “hoping it worked” into “knowing it worked.”
Reinforce: the mental model is the shared language between you and the agent. It's not extra work, it's leverage.
