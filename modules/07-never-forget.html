Module 7

Rules to never forget

The handful of habits and warnings that separate confident Git users from stressed ones.

This is the "if you remember nothing else" module. Practical safety, not theory.

Three habits prevent most pain

Commit small & often

One logical change per commit. Small commits are easy to understand and easy to undo.

Write real messages

“Fix login crash on empty email” beats “stuff”. Future-you reads these.

Pull before you start

Sync with the team first. It prevents most merge conflicts.

Small commits + honest messages + pull first = 90% of a smooth Git life.
These three habits prevent most pain. Commit granularity is the one people underdo.

Secrets and force-pushes are special risks

Never commit secrets. Passwords, API keys, tokens — once pushed, treat them as leaked even if you delete them later, because they stay in history.

Use a .gitignore file to tell Git which files to never track (keys, .env, large data, build junk).

Don’t force-push shared branches. git push --force can erase teammates’ work on main. If you ever need it, ask first and use it only on your own branch.
Secrets-in-history is the classic disaster. Force-push on shared main is the classic team disaster. Both are avoidable.

Undo by matching fix to situation

Almost nothing in Git is truly lost. Match the situation to the fix:

git restore <file>Throw away un-staged edits to a file — back to last commit.
git restore --staged <file>Un-stage something you added by mistake.
git commit --amendFix the last commit’s message or contents (before pushing).
git revert <commit>Safely undo an already-pushed commit by adding an opposite one.
Rule of thumb: revert for anything already shared (it’s safe); restore/amend for local, not-yet-pushed work.
revert vs reset: teach revert (safe, adds a commit) for shared history; avoid reset --hard for beginners.

Revert shared mistakes calmly

You (human) 
git log --oneline      # find the bad commit
git revert a1b2c3d      # undo it safely
Ask the agent

“My last commit broke things — revert it and explain what you did.”

The agent finds the commit and reverts it. You confirm the right one was undone.

When something breaks: don’t panic, don’t delete files. Ask “what changed?”, then revert.
The emotional message: git means mistakes are recoverable. Reverting is a normal daily action, not a failure.
